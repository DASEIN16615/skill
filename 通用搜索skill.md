# Deep Search — 通用深度搜索

## 触发条件

### 触发关键词（必须同时指向「外部网络信息」才触发）

- 中文：搜索、查一下、查查、帮我搜、查找、检索、搜一搜、网上找找、查资料
- 英文：search online、look it up、find information about、research this

### 排除规则（含以下意图时不触发，即使出现上述关键词）

- 本地文件/笔记操作："帮我找一下桌面上的文件"、"搜索我的笔记"、"查一下聊天记录"
- 内存/上下文操作："搜索一下你记得什么"、"找一下我们之前说过的"
- 代码/命令操作："搜索这个字符串"、"find 命令怎么用"
- 纯语言表达（非指令）："I find it interesting"、"research shows that"

判断原则：**意图是否指向「去互联网获取当前未知信息」**。如不确定，可先询问用户确认，不要盲目触发。

以上关键词列表不是穷举。用户可能用「了解一下 XX」「XX 是什么情况」「网上有没有关于 XX 的资料」等不含触发词的表达，只要意图明确指向外部信息获取，同样触发。

常见变体表达（不要死匹配关键词，这些也算触发）：
- "帮我了解一下 XX"
- "网上有没有 XX"
- "XX 是什么情况"
- "给我找找 XX"
- "现在 XX 怎么样了"
- 任何可以合理理解为「帮我上网查查」的表述

触发后，将用户的原始问题作为搜索主题，按以下流程执行。**不要在触发后立刻作答，必须完成搜索流程再输出。**

### 安装与激活

```bash
# 将 skill 目录放入 Hermes skills 文件夹
cp -r deep-search/ ~/.hermes/skills/

# 验证安装（应出现 deep-search 条目）
hermes skills list | grep deep-search
```

安装后 Hermes 会自动识别 description 中的触发说明，无需额外配置。

## 复杂度分级（触发后第一步判断）

触发后，先根据问题复杂度选择执行通道，不要无脑走完整流程：

| 级别 | 判断条件 | 执行路径 |
|------|---------|---------|
| **QUICK** | 单一客观事实查询，答案不存在争议（例："Python 最新版本"、"某国签证费用"） | web_search(1次, limit=3) → web_extract(1个最相关URL) → 直接输出，注明来源。不做多源交叉对比，不抓评论区 |
| **STANDARD** | 需要多角度信息，但不涉及主观评价或争议（例："考研408推荐教材"、"某城市旅行攻略"） | 走阶段 0-5，评论区仅抓主要平台 1 个 |
| **DEEP** | 涉及争议性判断、主观评价对比、或用户明确要求深度调查（例："XX 基金到底靠不靠谱"、"帮我深度调研 XX"） | 走完整阶段 0-5，所有平台评论区全抓 |

若不确定级别，默认 QUICK；若问题涉及主观判断或明显需要多源验证，升级为 STANDARD。

在开始搜索前，先判断问题属于哪个领域（可多选）：

| 领域代号 | 覆盖范围 |
|---------|---------|
| TECH | 编程、软件、硬件、AI、开源项目 |
| TRAVEL | 旅行规划、签证、景点、交通、住宿 |
| STUDY | 学习规划、教材选择、在线课程、备考 |
| KAOYAN | 考研信息、院校数据、专业课、政策 |
| STOCK | 股票、基金、ETF、宏观经济、投资策略 |

| NEWS | 时事热点、政策、社会事件、国际新闻 |

分类结果决定阶段 1 使用哪张权威源表。

### 1-B 动态权威源发现

在以下情况下，用 `web_search` 补充发现权威源，不受静态表限制：

1. 分类为 GENERAL
2. 静态表中的源返回结果不足 3 条
3. 静态表未覆盖某个具体子话题（如某个小众国家的旅行信息）

**动态发现方法：**
```
搜索词：[话题关键词] site:reddit.com OR site:github.com OR "[话题关键词] 官方"
```
提取被高频引用的域名，追加进本次搜索的来源池。

> 注意：动态发现的来源默认标记为「扩展来源」，在输出中注明。

## 阶段 2：正文内容提取

**工具选择原则：优先 web_extract，仅在失败时降级到 browser_navigate。**

### 执行模板

```
Step 1: web_search(query="[关键词] site:[权威域名]", limit=5)
        → 获取候选 URL 列表

Step 2: web_extract(urls=[url1, url2, ...])
        → 批量提取正文为 markdown（快速、低消耗）
        → 如果某个 URL 返回空或失败，记录「正文提取失败」，跳过
        → 失败计数：连续 3 个 URL 提取失败则停止当前来源池的重试，
          转入动态发现（1-B）补充新候选 URL，避免无限循环

Step 3（仅当 web_extract 失败或页面为动态渲染）:
        browser_navigate(url) → browser_snapshot()
        → 提取页面快照文本
        → 完成后直接滚动到评论区（复用此次 browser_navigate 会话，不重新导航）
```

### 从每个页面提取

1. **标题和发布时间** — 判断时效性
2. **正文核心论点** — 不要全文复制，提炼主张和关键数据
3. **作者/机构** — 判断来源属性（官方/媒体/个人/UGC）
4. **内部引用链接** — 如引用了其他权威来源，可顺藤摸瓜追加到来源池

**时效性判断规则：**
- 时事、股票、政策类：超过 30 天的内容需标注「可能过时」
- 旅行、考研类：超过 1 年的内容需标注「可能过时」
- 技术文档、学习资源：参考发布日期，差异较大时标注

## 阶段 3：评论区抓取

**仅在 STANDARD 和 DEEP 级别执行。QUICK 级别跳过本阶段。**

工具调用顺序：
```
browser_navigate(url)                     → 导航到目标页面（若阶段2已导航，直接复用会话跳过此步）
browser_scroll(direction=down)            → 滚动到评论区位置（通常需要多次）
browser_click("展开评论" / "查看更多")   → 若页面存在「展开」按钮，需先点击再 scroll（SPA 常见）
browser_snapshot()                        → 获取当前页面快照
→ 从快照中提取评论文本和点赞数
```

### 平台操作指南

**知乎**
- 滚动到回答底部，展开评论区
- 若有「查看全部 N 条评论」按钮，用 browser_click 点击后再 scroll
- 按「热度」排序（默认），提取点赞数 > 50 的评论
- 注意区分：回答本身（作者观点）vs 评论区（读者反馈）

**小红书**
- 展开笔记评论区，若有「展开更多评论」按钮，先 browser_click 再提取
- 提取点赞数前 5 的评论
- 标注：此平台 UGC 为主，存在种草/广告风险

**YouTube / B站**
- 按「热门」排序，提取 Top 5 评论
- 注意固定评论（UP主置顶）可能有利益关系，需标注

**Reddit**
- 提取每个帖子 Top 3 的 upvoted 评论
- 注意 subreddit 文化差异（如 r/wallstreetbets 与 r/investing 的风格迥异）

**微博 / 抖音评论**
- 提取热评前 5；微博部分内容需登录，遇到登录墙直接记录不可用
- 若有「展开」按钮，先 browser_click
- 标注：此类平台情绪化评论较多，主观体验仅供参考

**GitHub Issues / Discussions**
- 提取 reaction 数最高的回复（thumbs up 最多）
- 这类评论通常技术含量较高，可信度较其他平台更高

**Stack Overflow**
- 提取 accepted answer 和 score 最高的非 accepted answer
- 评论区中 score > 10 的评论也纳入

### 评论区失败回退策略

不要因为评论区抓取失败而卡住或重试超过 2 次。按以下顺序降级：

```
尝试 1: browser_navigate + browser_scroll×3 + browser_snapshot
   ↓ 失败（登录墙/验证码/超时 >15秒）
尝试 2: web_extract(url)  →  从静态页面提取可见评论（部分平台有 SEO 预渲染评论）
   ↓ 仍失败
记录为「[平台名] 评论区不可用：[原因]」，继续阶段 4，不阻塞流程
```

评论区不可用时，阶段 4 的交叉验证仅基于正文来源执行，在输出中注明「评论区数据缺失，交叉验证仅依赖多篇正文」。

## 阶段 4：交叉验证

对每一个**客观信息点**（数据、事实、政策），执行以下检验：

### 4-A 异议触发条件（需在输出中标注）

| 情况 | 标注方式 |
|------|---------|
| 正文结论与评论区主流反馈**相反** | `⚠️ 正文/评论存在分歧` |
| 同一客观信息在不同来源**数值不一致** | `⚠️ 数据存在矛盾` |
| 该信息**仅出现在单一来源** | `⚡ 单一来源，未经验证` |
| 来源为 UGC 平台且**无法找到一手来源** | `📌 UGC来源，建议核实` |
| 时间敏感信息**超过时效阈值** | `🕐 内容可能过时` |

### 4-B 主观信息处理

用户体验、个人意见、使用感受等主观内容：
- **不进行交叉验证**（主观信息没有对错）
- **全部呈现**，标注为 `💬 用户主观体验`
- 多个用户的主观信息可以汇总归纳，但保留代表性原话（简短引用即可）

### 4-C 不猜测原则

- 如果两个来源矛盾，**不要自行判断哪个正确**
- 标注矛盾后，列出两种说法，让用户自行判断或提供进一步核实建议
- 如果不确定某来源的可信度，标注疑问，**不要补全猜测**

## 执行约束

1. **不提前作答**：触发 deep-search 后，必须完成阶段 0-4 才能输出，不要在搜索过程中插入自己的判断
2. **不捏造评论**：如果评论区抓取失败，如实说明，不要用"用户普遍认为…"这类模糊表述替代
3. **不隐藏矛盾**：发现数据矛盾时，两种说法都要呈现，不选边站
4. **来源透明**：每条信息必须能溯源到具体 URL，不使用"有报道称"等匿名引用
5. **时效标注**：所有搜索结果注明访问时间（执行当天日期）

## Pitfalls（常见失败模式）

- **误触发**：用户说"你能搜索一下自己的记忆吗"——这不是 deep-search 触发，判断意图是否指向外部网络搜索
- **来源循环引用**：A 引用 B，B 引用 A，两者都不是一手来源——标注为 `⚡ 单一来源，未经验证`
- **评论区被广告污染**：小红书、抖音评论中植入广告较多，遇到疑似推广内容（带联系方式、引流链接）标注为 `📌 疑似推广内容`
- **Browser 工具超时**：某些页面加载慢，设定合理的等待时间（建议等待完整加载后再提取，最多等 15 秒）
- **数据源已下线**：页面 404 或内容已删除，记录为「来源不可访问」，不影响其他来源

## 兼容性说明（非 Hermes 环境）

本 skill 的 procedure 部分可直接复制到其他 AI 助手（Claude、GPT、Gemini 等）的 system prompt 或对话中使用。非 Hermes 环境下的工具映射：

| Hermes 工具 | 替代方案 |
|------------|---------|
| `web_search` | 该平台的搜索功能 |
| `web_extract` | 该平台的 URL 内容读取功能 |
| `browser_navigate / browser_scroll / browser_snapshot` | 该平台的浏览器自动化功能 |
| 无浏览器能力 | 评论区阶段跳过，输出中注明「当前环境不支持评论区抓取，交叉验证仅依赖正文来源」 |
